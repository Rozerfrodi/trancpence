
# Trancpence 

мини проект по учету доходов и расходов пользователя написан на DRF + React




## Features

- авторизация пользователя
- логирование действий пользоватля 
- возможность загружать файлы (выписки в форматье xlsx) для дальнейшего анализа
- Различные агреггированные данные для анализа финансов, трат за период и т.д.
- может что-то забыл)
## How work?
все модели (SQL таблицы) описаны в `models.py` каждого приложения


- в каталоге `files/`
находятся загруженные пользователем файлы, а также файл `example.xlsx` для отдачи его пользователю (см. `users/views.py` функция `get_example_file`)


- в каталоге `loan/`
Новое приложение по учету долгов и займов пользователя, будут 4 варианта (кредит, ипотека, расрочка, займ). С моей стороны реализовано создание объекта кредита в `LoanCRUDView`, также вывод списка текущих кредитов пользователя, удаление кредита, частичное изменение, вывод иконок для установки в карточку кредита (рисунок 1),
`LoanDetailOPS` будет отвечать за вложенные опреации в кредите (например пользователь будет добавлять то как он оплатил кредит (дата платежа, сумма, наименование)) эти операции можно будет удалять и т.п. (рисунок 2). Пока что не тестировали и приложение находится в разработке


- в каталоге `main/`
в файле `main/views.py`
собрана логика отображения ключевой информации по пользоватлю, `graph` выдает аггрегированные данные за переданную (период по дате, теги), `graph_ops` детальный список по операциям по (периоду, типу, теги, операции (расход/доход), сумме от/до, сортировка (desc/ascend)) за реализацию отвечает функция `param`, также `base_queryset` для сокращения написания кода для каждого из пунктов во `GraphViewSet` и `DetailViewSet` (в дальнейшем хочу переделать, т.к. считаю не данную реализацию идеальной)
`GetTagsAPIView` отвечает за получения списка существующих тегов. `GetTagsAPIView` отвечает за детальное отображения карточки операции по переданному `id`. `tpshka` просто дебаг функция для редиректа на документацию redoc. В файле `main/serializers.py` прописана логика по валидации полученых фильтров, для операций указаных выше. Также это было самое первое приложение в проекте  


- в каталоге `trancpence/`
тело проекта, собранны настройки django/drf настройки `celery tasks`, также подключены наборы эндпоинтов по трем приложениям (loan, users, main)


- в каталоге `users/`
в файле `users/views.py` в целом переопределены некоторые типовые методы для работы с авторизацией пользователей (именно необходимость указания файла при регистрации, настройка рассылки регистрациионого письма на почту и еще некоторые детали), также определен еще один @action`stats`для отображения статистики в профиле пользователя (кол. загруженных файлов, всего доходов/расходов, самый затратный месяц), `get_usergraph` небольшой @action для отрисовки финансовой динамики по месяцам за текущий год (на фронте отрисовывается график по этим данным)
`UserFileViewSet` отвечает за работу с файлами со стороны пользователя (добваление, удаление, просмотр списка файлов), 
`UserLogsViewSet` отвечает за отображение логов в профиле, также вывод по фильтрам и их очистку, в нем реализованы задачи через `celery`
`CompareViewSet` отдельный мини раздел для сравнения двух разных периодов между собой, с выведением советов по финансовому поведению и т.д., обработка идет через
`celery`. 
`TaskStatusView` так сказать глобальный класс (не совсем в том месте, ну это будет пересмотренно в дальнейшем) для получения результата задач с celery tasks, фронтенд при работе с `CompareViewSet` например в ответ получает ссылку на задачу и ожидает пока задача не будет завершена с результатом success, после чего делает get запрос по уникальному id задачи и получает результат выполнения задачи, потом этот ответ парсит.
`GetCurrencyAPIView` отвечает за получения списка валюты и ее установки у пользователя. Пока реализованно просто визуально, пока думаем как можно применить это на практике.


- также места где требуется пароли или другая информация например пароль от user в БД вынесены в файл `my.env`
